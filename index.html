AI Agent 導入支援の小型実証

Team Sub-3AI Agent活用の実践 ／ 業務変革への活用を見据えて

個人開発を、企業のAI Agent導入支援に近い「型」で運用するプロジェクト。 フルマラソン・サブ3（3時間切り）を目指すトレーニング管理アプリを題材に、 課題定義・Agent設計・データ設計・実装・運用改善を一気通貫で実証しています。

これは面接用のデモではありません。Garmin・食事・睡眠のデータを毎日記録し、AIの提案で実際に走り続けている「生きた運用」。 だから、机上では見えない改善点が日々の現場から立ち上がってきます。

複数のデータを統合し、判断基準を決め、続けられるUIと運用ルールに落とす——この構造は企業の業務変革と同じ。 それを、役割ごとに最適なAIをアサインし、人間と組んだチームで回しています。

実際、半年でサブ4（4時間切り）を達成。いまはサブ3へ向けて運用を続けており、 データが増えるほど判断は鋭くなり、改善のたびにアプリも走力も伸びていきます。

この「課題を定義し、AIを実装や分析に使い、人間が最終判断して運用に定着させる」型は、 そのまま企業のAI Agent導入支援・DX・PMOへ移せます。作れることより、現場で回し続けられることに価値があると考えています。

テーマは「AIは提案と実装を民主化するが、実行・定着・改善は自動化しない」。 AIに丸投げせず、人間の判断を残した統制可能な運用を設計しています。

なぜやるか

2026年3月、板橋Cityマラソンでサブ4（4時間切り）を達成。だが、ここからサブ3（3時間切り）までの道のりは遠い。残された時間は約1年——勢いや根性だけでは、とても届かない。

目標

2027年3月・板橋Cityマラソンでサブ3。夢ではなく、期日を切って達成すると決めた目標。だから道具にも妥協したくなかった。

現状

2025年10月にトレーニング開始。当初は「毎日走れば速くなる」という思い込みで、ただ漫然と走る日々。年末にウェアラブルを導入してもログを眺めるだけの"記録計"止まりで、「本当に速くなっているのか、完走できるのか」という漠然とした不安が常にあった。

課題

専門知識がない。専属コーチを雇うお金もない。無理なトレーニングは故障を招き、効率よく鍛えなければ残り1年では到底間に合わない。だから科学的手法で、最短距離のトレーニングを回したかった——その手段がAIだった。

解決策

自分の状態まで見て判断を変えてくれるコーチは、世の中にいない。なら、自分専用のAIコーチを作るしかない——そう決めたのだ。それが、いま毎日使っているこのアプリだ。

何を作るか

AIを"手段"として、毎日の意思決定を回している。作る → 課題を解く → 目標へ、の一本道。

1
何を作り、何を行っているか

その日の状態に応じて計画を変える意思決定アプリを自作。毎日 Garmin＋Omron＋食事＋主観コンディションを自動同期・記録し、AIが練習・食事・回復を提案。APIコストを抑える**ローカルLLM（Ollama Qwen3:8b）による評価経路**、不通時の**内製ロジックへの自動フォールバック**、および**常駐型ヘルス/バグ自動検知ポーラー**を組み込んだ堅牢な基盤で稼働しています。

2
それがどう課題を解決するか

足りない専門知識をAIが補い、雇えない専属コーチの代わりに24時間いつでも相談できる自分専用コーチになる。故障リスクを監視して"無理"を止め、限られた時間で最短距離のトレーニングを実現する。

3

科学的根拠に基づく日々の判断（練習・食事・回復）を、自己流に頼らず下せるようになった。

コーチを雇えない → AIが専属コーチを代替

専属コーチ相当の助言を24時間・コスト0で得られる体制になった。

1年で効率化必須 → 状態適応で最短化

状態に合わせ「やり過ぎ／不足」を抑え、故障リスクに配慮して最短距離で積み上げる。

Web・開発の専門ではない → AIにコーディングを任せる

エンジニアが本業ではない。実装はCodexなどのAIに任せ、自分は仕様定義と受入判断に集中して、フルスタックのアプリを形にした。

データ基盤の構築経験がない → AI支援で設計・実装

Garmin連携・Supabase・GASの構成をAIの支援で設計。データの取り込みから評価・可視化までを自前で組んだ。

どう運用するか

2つのフローを分離。毎日の運用（アプリを使う側）と、開発・改善（アプリを作る側）。

① 毎日の運用フロー日々の練習・食事を入力 → AIが判定 → 提案 → 実行 → 記録
② 開発・改善の仕組み（裏側）アイディアから本番反映まで、AIと人間で回す開発サイクル
🛡️ 開発プロセスを守る3つの機械ガードレール（Harness）
  • ① 自動コンテキスト注入（Session Start）: 起動時に前回の失敗教訓（失敗ログYAMLから自動抽出）と共通ルールをマージしたダイジェストをAIのコンテキストへ自動で読み込ませ、人間が必読する「善意依存」を排除。
  • ② プリデプロイ・ゲート（Pre-deploy Gate）: 本番反映（clasp push/deploy）の直前に git status や git diff --check をフックして機械検証し、未コミット変更やコンフリクトマーカーの混入を強制ブロック。
  • ③ 常駐監視バックアップと自動レビュー: ボードの書き込み監視（2秒間隔世代バックアップ）で消失事故を予防し、実装完了（確認・待ち）状態への遷移をタスクスケジューラで検知してClaudeがヘッドレスで自動静的レビューを実行。
